# Webhook payload format

Ryvos sends outbound HTTP webhooks in exactly one situation: when an
inbound request to `POST /api/hooks/wake` carries a `callback_url`
field and the underlying agent run has finished. This document
describes the payload schema, the retry behavior, and the
authentication model on the callback side.

Outbound webhooks are the notification half of the wake-and-notify
pipeline. The inbound half is documented in
[gateway-rest.md](gateway-rest.md#post-apihookswake) and uses a
separate token in `gateway.webhooks.token` for authentication. The
two halves together let an external system hand a prompt to Ryvos,
go back to its own work, and receive the result when Ryvos is done.

The implementation lives in the `webhook_wake` handler at
`crates/ryvos-gateway/src/routes.rs:329`. The callback is a
`reqwest::Client::post` call fired from a `tokio::spawn` inside the
wake handler after the agent run returns.

## When a callback fires

A callback fires only when both conditions hold:

1. The caller of `POST /api/hooks/wake` included a `callback_url`
   field in the request body. A request with no `callback_url` never
   produces an outbound webhook, even on error.
2. The agent run finished — either with a successful response or
   with an error. Both outcomes trigger the callback; only the
   payload shape differs.

If `callback_url` is present but the agent run is still executing
when the client disconnects, the callback still fires once the run
completes, because the wake handler holds a reference to the
runtime and awaits it before spawning the callback task.

No callback is sent if `gateway.webhooks.enabled` is `false` or if
the webhook token check fails — in those cases the wake request is
rejected with `404` or `401` before the runtime is invoked.

## Request shape

The callback is an HTTP `POST` to the exact `callback_url` supplied in
the wake request. The body is JSON with `Content-Type: application/json`
(set by `reqwest`'s `.json(&body)` call). There is no signing header,
no HMAC, and no retry. The request looks like this from the callback
server's perspective:

```bash
# What Ryvos sends to callback_url
POST https://ci.example.com/ryvos-done HTTP/1.1
Host: ci.example.com
Content-Type: application/json
Content-Length: <N>

{ "session_id": "...", "response": "...", "metadata": {...} }
```

The headers carry only `Host`, `Content-Type`, `Content-Length`, and
`User-Agent: reqwest/<version>`. Ryvos does not add any custom
headers, which means callback servers need to authenticate the
request on the URL path or on a query-string secret embedded in the
`callback_url` itself. See the authentication section below.

## Payload on success

On a successful agent run, the callback body is:

```json
{
  "session_id": "goal:20260410-123456",
  "response": "Build completed. 142 tests passed.",
  "metadata": {
    "project": "ryvos-core",
    "commit": "a1b2c3d"
  }
}
```

Fields:

| Field | Type | Description |
|---|---|---|
| `session_id` | string | The session the run executed in. If the wake request supplied an explicit `session_id`, this matches it; otherwise it is the random session ID generated by `SessionId::new`. |
| `response` | string | The final agent response, exactly as returned by `AgentRuntime::run`. |
| `metadata` | any | The `metadata` field from the original wake request, passed through unchanged. Any JSON value is accepted: object, array, string, number, bool, or null. Omitted entirely when the wake request did not include metadata. |

## Payload on error

When the agent run fails, the callback body replaces `response` with
`error`:

```json
{
  "session_id": "goal:20260410-123456",
  "error": "context length exceeded",
  "metadata": {
    "project": "ryvos-core",
    "commit": "a1b2c3d"
  }
}
```

Fields:

| Field | Type | Description |
|---|---|---|
| `session_id` | string | Same as the success case. |
| `error` | string | The `Display` form of the `RyvosError` returned by `AgentRuntime::run`. |
| `metadata` | any | Pass-through from the wake request. |

A callback server should treat the presence of `error` rather than
`response` as the discriminator, not the HTTP status of the incoming
callback (which is always `200` from Ryvos regardless of agent
outcome).

## Authentication

Ryvos sends the callback as an unauthenticated POST. There is no
Bearer header, no HMAC signature header, and no shared secret in the
body. The operator is responsible for securing the callback URL
through one of the following patterns:

- **Embed a secret in the URL.** Generate a random token per
  integration and include it as a path segment
  (`/webhooks/ryvos/abc123`) or a query parameter
  (`?token=abc123`). The wake caller sets the full URL; Ryvos sends
  the POST to it verbatim. This is the recommended pattern and the
  shortest path to a working integration.
- **Put the callback endpoint behind a reverse proxy with allowlist
  rules.** If the callback server is internal, restrict inbound
  traffic to the host running Ryvos by IP.
- **Accept that the callback is public and treat the payload as
  untrusted.** Validate `session_id` against a list of sessions the
  caller started, and drop callbacks for sessions you do not
  recognize. This works when the callback consumer is idempotent
  and the blast radius of a spurious delivery is small.

None of these patterns is implemented by Ryvos — they are conventions
for whoever runs the callback server. The guidance is listed in
order of preference.

## Retry behavior

Callback delivery is fire-and-forget. The wake handler spawns a
`tokio::task` for the outbound POST and returns to its caller
immediately. If the POST fails (DNS error, connection refused, TLS
handshake failure, non-2xx response, timeout), the failure is logged
with `tracing::warn!` and the task exits. There is no retry, no dead
letter queue, and no persistence. The sequence from the handler looks
like this:

```rust
// crates/ryvos-gateway/src/routes.rs:380
tokio::spawn(async move {
    if let Err(e) = client.post(&url).json(&cb_body).send().await {
        tracing::warn!(url = %url, error = %e, "Webhook callback failed");
    }
});
```

Practical consequences:

- A callback server that is down when Ryvos fires the POST will miss
  the delivery. The wake caller can poll `GET /api/runs` to
  reconstruct the outcome from the cost store, or simply trigger
  the wake again.
- A callback server that responds slowly does not block the wake
  response back to the original caller, because the task is
  spawned. It does consume a Tokio task until the POST finishes or
  the configured `reqwest` timeout fires.
- There is no deduplication. A wake request that fires twice fires
  two callbacks.

Callers that need at-least-once semantics with retry should run their
own queue in front of the callback handler and poll
`GET /api/runs` rather than relying on the outbound webhook.

## Common use cases

- **CI pipelines.** A GitHub Action hits `POST /api/hooks/wake` with
  a prompt and a `callback_url` pointing at a status endpoint; the
  `metadata` field carries the commit SHA and PR number for
  correlation.
- **Chat bots.** A Slack slash command forwards the user message
  with `callback_url` pointing at a `chat.postMessage` relay.
- **Pipeline chaining.** An upstream workflow in n8n, Make.com, or
  Zapier fires Ryvos as one step and resumes the pipeline when the
  callback arrives.

All three patterns use `metadata` as a correlation key between the
wake request and the callback, which sidesteps the lack of a
dedicated request ID on the wire.
